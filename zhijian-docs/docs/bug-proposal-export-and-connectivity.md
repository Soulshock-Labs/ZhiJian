# Bug 提案：导出失败 / 后端连接误判 / 用户无法获知真实原因

| 字段 | 内容 |
|------|------|
| **产品** | 智伴幼师 smart-teacher-assistant |
| **类型** | Bug + UX 缺陷（多因素叠加） |
| **严重程度** | P1（核心路径「生成并下载 Word」不可用或不可感知） |
| **状态** | 部分已在代码中修复（见 §5），待回归验证 |

---

## 1. 问题摘要

用户在多种使用方式下（Cloud Run HTTPS、本地 `localhost:8000`、本地双击 `index.html`）反馈 **「无法导出」** 或 **「三种方式都不行」**。  
表象为：无 `.docx` 下载、或仅有网页预览、或无任何明确错误说明。

---

## 2. 影响范围

- **主流程**：第二步上传模板 →「把灵感装入模板」→ 期望自动下载填充后的 Word。
- **受影响用户**：
  - 使用 **`file://` 打开本地 `index.html`** 且 `API_BASE` 为空时（相对路径请求落到 `file://`，健康检查必失败）。
  - **未上传 `.docx`** 仍点击生成时（进入演示模式，**无下载**但此前**无明确弹窗**）。
  - **Safari / 部分移动浏览器 / 微信内置浏览器**：异步 `fetch` 完成后的**脚本触发下载**常被拦截，用户误以为「没导出」。
  - **无法访问 Google Cloud（网络/地区）**：`/health` 失败 → 演示模式，原因不透明。
- **周日联动**：`/generate-daily` 存在同类下载与后端依赖问题。

---

## 3. 复现步骤（示例）

### 场景 A：本地双击 HTML

1. 用资源管理器 / Finder **双击** `index.html`（地址栏为 `file://...`）。
2. 完成主题与理念填写，**已上传** `.docx`，点击生成。
3. **旧版行为**：`API_BASE === ''` → `fetch('/health')` 实际请求 `file:///health` → `checkBackend()` 为 false → **静默进入演示模式**，无 Word 下载。

### 场景 B：未上传模板

1. 在第二步 **未上传** Word 模板（或上传后刷新导致状态丢失）。
2. 点击生成。
3. **旧版行为**：`!st._fileObject` → 演示模式，**无弹窗说明「必须先上传模板」**。

### 场景 C：Safari

1. 使用 Safari，后端与模板均正常，`/generate` 返回 200。
2. **旧版行为**：自动 `a.click()` 下载被拦截，用户看不到文件，也**无备用下载入口**。

---

## 4. 期望行为 vs 实际行为（修复前）

| 维度 | 期望 | 实际（修复前） |
|------|------|----------------|
| 后端不可达 | 明确提示「无法连接 API」及建议地址 | 进入演示模式，易被理解为「导出坏了」 |
| 未上传模板 | 明确提示必须先上传 `.docx` | 静默演示模式 |
| 下载被拦截 | 提供可点击的备用链接（同一次生成的 Blob） | 仅自动触发一次，无兜底 |
| `file://` + 线上 API | 应能指向合法 HTTPS API 基址 | `API_BASE` 为空，请求错误 |
| 响应头可观测性 | 可知实际填充引擎（Aspose / python-docx） | 曾固定为 `aspose-words`（已另案修复） |

---

## 5. 根因分析

1. **前端 `API_BASE` 策略缺陷**：非 `localhost` 且非 `file:` 时用相对路径；**`file://` 下相对路径无效**，导致健康检查与 `/generate` 均失败。
2. **条件分支合并**：`!backendOK || !st._fileObject` 共用同一条「演示模式」路径，**未区分原因**，用户无法自查。
3. **浏览器安全策略**：下载需用户手势链；**异步完成后程序化 `click` 下载**在 Safari 等环境不可靠。
4. **（次要）** CORS：`allow_origins=["*"]` 与 `allow_credentials=True` 并存不符合规范，极端环境下跨域行为难预测（已建议改为 `allow_credentials=False`）。

---

## 6. 已实施 / 建议修复清单（实现侧）

| 序号 | 措施 | 说明 |
|------|------|------|
| F1 | `file://` 默认 `PRODUCTION_API_DEFAULT` | 指向 Cloud Run 等明确 HTTPS 基址 |
| F2 | `?api=`、`localStorage STA_API_BASE` | 可覆盖 API 地址 |
| F3 | 拆分提示 | 未上传模板 vs 连不上后端 **分别 `alert` + 日志** |
| F4 | 备用下载 UI | 结果页展示「没有自动弹出下载？」+ `<a download>` 链接触发同一 Blob |
| F5 | `fakeDownload` / `_lastWordBlobUrl` | 支持「再次导出」同一份文件 |
| F6 | `checkBackend` 超时 | `AbortController` 替代 `AbortSignal.timeout` 以兼容旧浏览器 |
| F7 | 极小响应体校验 | `blob.size < 80` 时抛出可读错误，引导看 Network |
| F8 | 后端 CORS | `allow_credentials=False`（若前端不依赖 Cookie） |
| F9 | 导出引擎响应头 | `X-Export-Engine` 与实际 Aspose / python-docx 一致（另项） |
| F10 | 净空失败回退原始 docx | 避免个别模板导致整单 500（另项） |

## 7. 验收标准（建议 QA 勾选）

- [ ] `https://<部署域名>/` 打开：上传 `.docx` 后生成，**能下载**或**能通过备用链接下载**。
- [ ] 本地 `file://` 打开**更新后的** `index.html`：已配置默认云端 API 时，`/health` 成功，**能生成并下载**（或明确跨域失败提示）。
- [ ] **不上传模板**点击生成：出现**明确中文说明**，而非仅演示预览。
- [ ] **拔网线或错误 `API_BASE`**：出现**连接后端失败**类说明，而非与「未上传模板」混淆。
- [ ] Safari：即使自动下载失败，用户仍可通过**页面上的下载链接**保存 `.docx`。
- [ ] Network：`/generate` 200 且 `Content-Type` 为 Word 相关；失败时前端展示可读 `detail`（含数组型校验错误）。

---

## 8. 后续建议（非阻塞）

- 在第二步模板区增加**常驻短文案**：「未上传模板则不会生成 Word 文件」。
- 部署页首屏脚注：推荐使用的浏览器（Chrome / Edge）及「勿仅用微信内置浏览器完成下载」。
- 监控：Cloud Run 日志中 Aspose 回退 `warning`，便于区分格式问题与纯网络问题。

---

## 9. 附录：与用户沟通时可用的简短说明

> 导出依赖三件事：**能连上 API**、**已上传 .docx**、**浏览器允许下载**（Safari 请用页面上的「下载 .docx」链接）。  
> 仅网页预览、没有文件，多半是未上传模板或后端未连通，不是「Word 生成功能单独坏了」。
