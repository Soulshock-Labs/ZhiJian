## 2. 背景与目标

### 2.1 背景

- 园所使用的模板来源不统一，用户可能**并非基于「上一周」模板**继续填写，系统无法可靠假设「上周文档」存在或一致。
- 市面缺乏能真正解决该场景痛点的方案；产品价值在于**解决具体业务痛点**，而非表格是否「标准版」。

### 2.2 目标

1. **业务目标**：支持教师高效完成**本周周计划**，并可在认可价值的前提下，**按天生成与周计划一致的每日教案**。
2. **体验目标**：在信息可配置的前提下，尽可能提供**傻瓜式、一键式**生成路径，降低操作负担。
3. **商业目标**：通过可感知的生成价值驱动 **Token/用量消耗**，与订阅定价（如月费/年费区间）匹配。

## 3. 术语

| 术语 | 含义 |
|------|------|
| 周计划 | 以周为单位的教学/活动安排文档 |
| 每日教案 | 从周计划拆解到「周一～周五」单日的教案内容 |
| 班型 | 小班 / 中班 / 大班（与年龄段相关） |
| 集体活动 | 园所日程中的一类活动；数量与「动静结合」规则随班型变化 |
| Token | 生成类能力消耗的计量单位（对话中称「头粉」） |

---

## 4. 用户与场景

### 4.1 主要用户

- **一线教师**：填写周计划、需要生成或调整每日教案；可能关注表格美观与可用性。
- **产品/研发**：在「用户真实场景」与「可实现的智能程度」之间对齐需求。

### 4.2 典型场景

1. 用户上传或基于**任意模板**开始工作，需要明确**第几周**、**具体日期**。
2. 完成周计划后，希望按**周一、周二……**逐日生成教案，且**与当周计划强一致**（不可跑题、不可与本周安排无关）。
3. 园所班型不同（小班/中班/大班），**集体活动**节数与排课约束不同，需要系统辅助遵守。

## 5. 功能需求

### 5.1 周次、日期与「本周」语义

| ID | 需求描述 | 优先级 | 备注 |
|----|----------|--------|------|
| F-1.1 | 用户可选择或指定**第几周**，而非隐含依赖「上一周模板」。 | P0 | 与「模板来源混乱」现实一致 |
| F-1.2 | 支持选择或展示**具体日期**（与「本周」语义一致）。 | P0 | 纪要中列为待跟进点之一 |
| F-1.3 | 「生成本周」类能力应以用户**当前提交的模板与内容**为基准，**不能**强制以「上周」为唯一标准。 | P0 | 业务规则，非可选优化 |
| F-1.4 | 在可配置项增加时，评估对**操作步骤**的影响；理想路径仍为一键生成。 | P1 | 需 UX 设计权衡 |

**待决议（来自会议）**：周次/日期若全部手选，可能增加工作量；是否与「一键」目标冲突，需产品定方案（默认值、智能推断、可选高级设置等）。

---

### 5.2 班型与集体活动规则

| ID | 需求描述 | 优先级 | 备注 |
|----|----------|--------|------|
| F-2.1 | 系统应理解或配置**班型**（小班 / 中班 / 大班）。小班约对应**四岁以下**等园所常用划分（具体定义可配置）。 | P0 | 纪要：年龄段差异 |
| F-2.2 | **小班**：一天仅 **1** 个集体活动；需满足**动静结合**。 | P0 | |
| F-2.3 | **中班、大班**：一天 **2** 个集体活动；需满足**动静结合**。 | P0 | |
| F-2.4 | 排课约束示例（需落地为可配置规则或策略）：**不能**安排两节均为高操作性强、负担重的组合（如两节科学课；或「艺术 + 数学」类组合等，具体科目名以园所词典为准）。 | P1 | 纪要中为举例，需教研确认 |

---

### 5.3 周计划 → 每日教案

| ID | 需求描述 | 优先级 | 备注 |
|----|----------|--------|------|
| F-3.1 | 在**周计划已完成**的前提下，支持基于周计划生成**每日教案**。 | P0 | 核心链路 |
| F-3.2 | 每日教案须与**对应星期**的周计划内容**强绑定**：不可偏离本周主题安排、不可生成与本周无关的内容。 | P0 | |
| F-3.3 | 支持按**单天**选择生成（如仅周一、仅周二），而非必须一次性生成五天。 | P0 | 与计费、体验均相关 |
| F-3.4 | 生成节奏：**逐日生成**（今天生成周一、明日生成周二）优于「一次倒出五天」的交互预期（纪要中开发侧关注点）。 | P1 | 产品需定义默认交互 |

**说明**：是否允许用户完全手写某日教案而不走生成，纪要中存在讨论；**一致结论**为：核心仍是「与周计划对齐」；若手写，也应满足与周计划一致性的校验或引导（细节待补）。

---

### 5.4 模板、表格样式与园所差异

| ID | 需求描述 | 优先级 | 备注 |
|----|----------|--------|------|
| F-4.1 | 支持园所**非标准模板**导入或沿用现有版式；**不把「统一成标准表」作为产品目标**，否则削弱差异化价值。 | P0 | 产品定位 |
| F-4.2 | 表格**样式**需尽量贴近用户现有材料（纪要中称难点之一）。 | P0 | 与「美观」并存 |
| F-4.3 | 生成内容的**语气/风格**需贴近用户期望（纪要：「像你写出来的一样」为难点）。 | P1 | 偏 NLG/模板策略 |
| F-4.4 | 支持园所 **Logo** 等简单品牌化元素（纪要评估为相对简单）。 | P2 | |

| ID | 需求描述 | 优先级 | 备注 |
|----|----------|--------|------|
| F-5.1 | **按次/按天**的生成消耗 **Token**；用户需感知到产出价值才愿意付费消耗。 | P0 | |

## 6. 非功能需求

| ID | 类别 | 描述 |
|----|------|------|
| NF-1 | 可扩展性 | 多个园所逻辑类似，底层代码应可复用；先验证单园所闭环再复制。 |
| NF-2 | 可维护性 | 班型、集体活动规则、课程类型禁忌等宜**配置化**，避免硬编码。 |
| NF-3 | 智能性 | 「修改意见」类交互偏累人；长期需减少对用户的反复追问，**依赖更完整的需求输入或场景学习**（纪要中方向性描述）。 |

---

## 7. 约束与产品原则

1. **基准数据**：以用户**当下提交的模板与内容**为生成基准，不假设完整、干净的「上周」数据。
2. **差异化**：**不做**「全行业一张标准表」的替代方案；价值在可用、贴合园所现状。
3. **一致性**：周计划与每日教案之间必须**可追溯、可对齐**。

## 8. 待确认项与风险

| 类型 | 内容 |
|------|------|
| 待确认 | 周次/日期在「一键」与「可选」之间的默认策略；一次性导出五天是否保留为可选能力。 |
| 待确认 | 「艺术 + 数学」等排课禁忌的正式规则表与园所差异。 |
| 待确认 | 手写教案与 AI 生成混用时的校验规则与产品流程。 |
| 风险 | 需求若不能结构化传达给研发，输出「全智能」仍困难；需建立稳定的需求沟通机制。 |
| 风险 | Token 与订阅权益的打包方式影响用户感知，需单独做计费 PRD。 |
